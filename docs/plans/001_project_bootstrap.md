# Plan: Parallax Project Bootstrap

## Context
Parallax is a greenfield scientific augmentation tool leveraging agentic AI for robust, reproducible science. Currently only `parallax_idea.md` exists. This session bootstraps the project: structured design docs, developer CLAUDE.md, private GitHub repo, and project skeleton.

## Architecture: 3-Layer Model

### Layer 1: Convention System (MVP-alpha)
Maps heavily to Claude Code native primitives:

**Custom Python code:**
- `parallax init` — structured interview CLI that generates all config files below

**Claude Code skills (`.claude/skills/`):**
- `/hypothesis` — hypothesis workflow (ask human first → AI generates N alternatives → rank → test → conclude)
- `/handoff` — generate structured agent handoff summary
- `/audit` — manual project audit (code review, test gaps, concerns)
- `/experiment` — create new experiment manifest from template

**Claude Code hooks (`.claude/settings.json`):**
- `PreToolUse` on Edit/Write → block test tolerance reductions without human approval
- `PreToolUse` on Bash (git commit) → trigger auto-doc generation, verify tests pass
- `PostToolUse` on Edit/Write → lint/format
- `Stop` → enforce summary output, check if docs need updating
- `SessionStart` → inject constitution principles after context compaction

**Template files:**
- CLAUDE.md (project-specific, generated by interview)
- PARALLAX.md (scientific workflow rules, hypothesis protocols, agent handoff templates)
- CONSTITUTION.md (core scientific principles — "the values all agents internalize")
- Experiment manifest template (YAML)
- Hypothesis doc template
- Agent summary template

**Hard-coded baseline rules for all generated CLAUDE.md/PARALLAX.md** (adaptable over time):
- Concise interactions & commits; sacrifice grammar for concision except when clarifying code design/accuracy
- Loud errors over hidden errors/edge-cases
- Sanity checks on numerical units / dimensional analysis before finalizing code
- Plans must end with unresolved questions list (precise, concise)
- These are defaults; user interview can override any of them

**Recommended agentic workflow (encoded in PARALLAX.md):**
- Phase 1: Research/audit with Opus (deep reasoning, read-only)
- Phase 2: Hand off research notes to Plan phase with Opus in fresh context
- Phase 3: Implement with fresh context on Opus or Sonnet (Sonnet for simpler tasks)
- Each phase handoff includes structured summary to mitigate context rot
- This is a recommendation, not enforced — user can override

**CI/CD (GitHub Actions) — enforces rules at the infrastructure level:**
- Ruff (format + lint)
- mypy strict
- pytest (unit + regression)
- Semantic version validation (tags match pyproject.toml)
- Doc staleness check (are auto-generated docs up to date?)
- CI is the hard enforcement layer — hooks are advisory/local, CI is the gate

### Layer 2: State + Workflow Engine (MVP-beta)
- Hypothesis lifecycle tracking (proposed → active → tested → concluded) in SQLite
- **Parallel hypothesis exploration via git worktrees** — each hypothesis gets its own worktree from same tagged commit, tested independently, results compared. Priority ordering agreed between user & agents.
- Agent handoff summaries w/ context rot mitigation
- Test result history + regression tracking (SQLite)
- Semantic versioning automation tied to experiments
- Environment reproducibility (pixi/conda-lock)
- Conversation/session logging w/ daily summaries
- Auto-documentation system (API docs, quickstart notebook, README updates before commits)
- Failure/negative result logging (disproven hypotheses are scientifically valuable)
- **Basic versions of Layer 3 features**: simple token usage tracking, manual-trigger audits

### Layer 3: Full Orchestrator + Dashboard (v1+)
- Optimized multi-agent coordination w/ token budget management
- Automated scheduled audits
- TUI/web dashboard
- Visual time series of regression metrics
- JupyterLab hub integration
- Literature/DOI integration
- Provenance chains (result → code version → data version → environment → hypothesis)
- Data versioning (DVC/git-lfs integration)
- Multi-language support (Fortran, R, Julia, C++)
- Fun scientist on startup
- Voice interface

## CONSTITUTION.md Concept

Distinct from CLAUDE.md (build conventions) and PARALLAX.md (workflow procedures). The constitution encodes *values and decision-making principles* that all agents internalize:

- Robust science over fast science — accelerate as fast as possible but no faster
- NEVER weaken tests to pass — loud failure over silent success
- Hypotheses before implementation — think before you code
- Reproducibility is non-negotiable
- Human judgment drives scientific decisions; AI augments, never replaces
- Negative results are valuable — document disproven hypotheses with equal rigor
- Correctness and precision over feature velocity
- Loud errors over silent failures

Loaded into context via `@CONSTITUTION.md` import in CLAUDE.md. Reinforced via `SessionStart` hook after context compaction.

## JAX / Differentiable Code & Performance Backend Rules

Added to `parallax init` interview as configurable options:

**If user selects JAX/differentiable programming:**
- PARALLAX.md gets rules: prefer JAX-compatible ops, warn loudly when introducing non-differentiable ops
- Explicit flags when Rust/C extensions might break a differentiable chain
- Clear separation of differentiable vs non-differentiable code paths
- Recommendation hierarchy: JAX → NumPy+Numba → Rust/C (escalate only when needed, always ask)

**Performance backend escalation:**
- Python/JAX first → Numba JIT if needed → Rust/C for truly critical paths
- pixi manages mixed Python+Rust toolchains
- Always communicate to user when crossing language boundaries
- Rules encoded in PARALLAX.md, enforced via hooks where possible

## Auto-Documentation System (Layer 1 light, Layer 2 full)

**Layer 1 (hooks-based):**
- `PreToolUse` hook on Bash matching `git commit` → runs doc-check script
- Script checks if key docs (README, API docs, quickstart) are stale relative to code changes
- Warns (doesn't block) if docs may need updating

**Layer 2 (full system):**
- Auto-generate API docs from docstrings/type hints before each commit
- Quickstart.md that is convertible to Jupyter notebook (via jupytext or similar)
- README.md auto-updates installation instructions, feature list, etc.
- All auto-generated sections clearly marked so manual content isn't overwritten
